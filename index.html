Software Development Life Cycle

Semester:4thsemester
Subject:Software Engineering


The Software Development Life Cycle (SDLC) is a structured process used in software engineering to guide the development of high-quality software.
SDLC specifies the task(s) to be performed at various stages by a software engineer or developer. It ensures that the end product is able to meet the customer's expectations and fits within the overall budget. Hence, it's vital for a software developer to have prior knowledge of this software development process.



    Key Phases of SDLC :
  1. Requirements Gathering:

    the process of identifying, documenting, and managing the needs and expectations of stakeholders for a project or system

  2. System Design:

    Creating the architecture, user interface, and database design.

  3. Implementation:

    the phase where the software design is translated into a functional product through coding, testing, and integration

  4. Testing

    Evaluating the software for functionality, performance, and security.

  5. Deployment:

    Releasing the software to the intended environment.

  6. Maintenance:

    Ongoing support, bug fixing, and updates to the software.

SDLC Detailed Explanation:

1.Requirements Gathering:

Types of Requirements:


Functional Requirements:
Describe what the software should do (e.g., "The system shall allow users to create accounts.").
They describe the specific functions, features, and behaviors that the software must perform to meet the user's needs and achieve the desired outcomes.
Non-Functional Requirements:
Describe how the software should perform (e.g., "The system shall respond to user requests within 2 seconds.").
These can be further categorized into:
* Performance: Speed, scalability, reliability.
* Security: Protection of data and system.
* Usability: Ease of use and user experience.
* Maintainability: Ease of modifying and updating the software.
* Portability: Ability to run on different platforms.
Key Activities:
The requirements analysis phase typically consist of four basic activities.
• Requirement Elicitation
• Requirement Analysis
• Requirements Documentation
• Requirements Validation
Requirement Elicitation: Gathering information from various sources:
Interviews: Talking to stakeholders directly.
Surveys: Reaching a wider audience.
Workshops: Collaborative sessions to brainstorm and refine.
Document Analysis: Reviewing existing materials.
Prototyping: Creating early mock-ups to get feedback.
Requirement Analysis: Clearly understand the user requirements:  Identifying patterns and trends.  Resolving conflicts and inconsistencies.
Prioritizing requirements based on importance.
Modeling techniques are used to analyse the requirements.
Modeling: Representing the requirements in different ways:
Use Case Diagrams: Showing how users interact with the system.
User Stories: Short, user-focused descriptions of functionality.
Flowcharts: Visualizing processes and workflows.
Requirement Documentation:
After analyzing the requirements, requirements are systematically documented.
• Requirements Document is called Software Requirements Specification (SRS).
Software Requirements Specification • The SRS document is useful in various contexts:
• Forms an agreement between the customers and the developers
• Reduces future reworks

2.System Design :

Types of Design:


UML Diagrams and ER Diagram

3.Implementation :

This is where the actual coding takes place. Developers write the software according to the design specifications, using chosen programming languages and technologies.


4.Testing:

Types of Testing:


This includes unit testing, integration testing, system testing, and user acceptance testing


5.Deployment:

Once the software passes all testing phases, it's deployed to the production environment. This may involve installing it on servers, configuring it for users, and migrating data.

6.Maintenance:

After deployment, the software needs ongoing maintenance and support. This includes bug fixes, security updates, performance improvements, and adding new features based on user feedback.

Comparison table of SDLC Models:

MODEL NAME BEST USE CASE DRAWBACKS
Waterfall Clear, well-defined requirements; stable project; small to medium-sized projects; projects with minimal changes expected Inflexible; difficult to accommodate changes; testing is done late; limited client involvement; no prototyping
Agile Projects with evolving requirements; frequent changes; client collaboration is crucial; large, complex projects Requires strong communication and collaboration; can be challenging to manage; may lead to scope creep; requires skilled team
Spiral Projects with high uncertainty, high risk, and complex requirements; projects that need ongoing development and enhancement; frequent changes expected. Can be expensive and time-consuming due to frequent iterations; requires significant risk analysis expertise; can be difficult to manage.

Conclusion:

In conclusion, the Software Development Life Cycle provides a vital framework for organizing and managing the complex process of creating software. It ensures that projects are executed in a systematic manner, leading to higher quality products and more efficient development. Choosing the right SDLC model is paramount as it directly impacts project success, adapting to the specific needs and constraints of each software endeavor. A well-chosen model streamlines workflows, facilitates communication, and ultimately delivers robust and reliable software solutions.
